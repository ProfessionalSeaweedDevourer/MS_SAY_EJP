# Academy 백업 리포지토리 공개 정비 작업 지시문

## Context

박주현(Eric J. Park)의 MS AI SW Academy(Microsoft × 성균관대 K-Digital Training, 2025.10–2026.04, 6개월) 이수 기록 백업 리포지토리를 **공개 포트폴리오 보조 자산**으로 정비한다. 원래 private였으나 채용 과정에서 이력서·포트폴리오의 "MS AI SW Academy 6개 모듈 이수" 및 "Azure AI-900 취득" 항목의 근거로 링크할 예정.

## 독자와 목적

**주 독자**: 한국 및 해외 IT 기업의 채용담당자·면접관. 이력서·포트폴리오 링크를 타고 들어와 **10~30초 안에 "실제로 이 교육과정을 수료했고 학습의 질이 있다"**를 확인하려는 사람. 10분을 쓸 독자가 아님.

**부차적 독자**: Eric 본인(향후 재사용), Academy 동료, 관심 있는 개발자.

**이 리포지토리가 달성해야 할 것**
1. 6개 모듈 전체가 학습된 흔적이 **검증 가능한 형태**로 제시됨
2. 외부 독자가 **30초 안에 네비게이션** 가능한 구조
3. 우수 결과물이 **앞으로 끌어내 가시화**됨
4. 민감정보·저작권·수업 자료 원본 등 **공개 부적합 요소가 0건**
5. Azure AI-900 자격증·수료 증빙과 **연결 가능한 상태**

## Phase 1 — 실태 조사 (수정 금지, 승인 대기)

즉시 수행한 뒤 중단한다.

### 작업 내용

1. 리포지토리 루트부터 재귀적으로 스캔. 디렉토리 구조·파일 목록을 `PLAN.md`에 기록
2. 각 최상위 디렉토리에 무엇이 있는지 **1문장 요약**
3. 6개 모듈 중 현재 무엇이 있고 무엇이 없는지 체크리스트로 정리:
   - ① AI 기초 프로그래밍 (Python · SQL · MongoDB)
   - ② 훈련용 데이터 구축 (웹 크롤링 · 공공 데이터 수집)
   - ③ Full-stack 웹 개발 (HTML/CSS/JS → Flask → React+Python → FastAPI+MySQL)
   - ④ Azure 인프라 (VM · Linux · Cloud)
   - ⑤ AI 기초 이론 (통계 · CNN · RNN · GAN · KNN)
   - ⑥ MS Azure AI (Copilot · OpenAI · LLM · Vision · NLP)
4. 문제점 분류:
   - **Critical**: 민감정보(`.env`·API 키·학번·연락처·주민번호), 저작권 의심 자료(수업 슬라이드·교재 스캔본·강사 PPT·시험 문제), 대용량 바이너리(모델 체크포인트, 100MB+)
   - **High**: 구조 혼란(6개 모듈에 매핑 불가능한 위치), 실행 불가능한 코드, 의존성 명세 없음
   - **Medium**: README 없음·미완성, 한·영 혼재 일관성 부재, 주석 없는 raw 노트북
   - **Low**: 타이포, 파일명 규칙 불일치

### Phase 1 산출물

`PLAN.md` 파일 하나. 포함 사항:
- 현재 디렉토리 트리 (3 depth까지)
- 6개 모듈 매핑 현황표
- 문제점 분류 리스트 (각 항목 파일 경로 포함)
- Phase 2 작업 제안

**여기서 중단. 사용자 승인 전까지 어떤 파일도 이동·삭제·수정하지 않는다.**

## Phase 2 — 정비 실행 (사용자 승인 후)

### 작업 순서

1. **Critical 건 선처리**: 민감정보는 `.gitignore` 추가 + 현재 파일 삭제. history에 남아 있는 경우 `git filter-repo` 필요성을 사용자에게 별도 보고 (자동 실행 금지)
2. **저작권 의심 자료**: 직접 삭제하지 말고 파일 목록을 제시해 사용자 판단 받는다. 판단 전까지 해당 파일은 건드리지 않음
3. **6개 모듈 디렉토리 구조 정리**:
```
   modules/
     01-foundations/
     02-data-ingestion/
     03-fullstack/
     04-azure-infra/
     05-ai-theory/
     06-azure-ai/
```
   기존 파일은 `git mv`로 이동하여 history 보존
4. **각 모듈 README.md 신규 생성**. 템플릿:
```markdown
   # Module X — [모듈명]

   **기간**: YYYY-MM-DD ~ YYYY-MM-DD
   **핵심 개념**: [쉼표로 5~8개]
   **스택**: [쉼표로]

   ## 학습 범위
   - [bullet 3~5개, 커리큘럼 기준]

   ## 주요 결과물
   - `notebook-01.ipynb` — [한 줄]
   - `project-a/` — [한 줄]

   ## 참고
   - [관련 링크·자격증·후속 프로젝트]
```
   **공백 채우기 금지**. 파일이 없으면 "개인 노트는 별도 보관" 등 사실만 기재.
5. **노트북 처리**: 실행 결과(output)를 원칙적으로 **보존**한다. 이유: 독자가 실행하지 않고도 결과를 확인할 수 있어야 함. `nbstripout`으로 전체 output을 날리지 않음. 단, 다음 output만 제거:
   - API 키·토큰 에코
   - 개인정보 (이름·이메일·학번)
   - 수천 행 이상 데이터 덤프 (앞 10행만 남기고 `...` 처리)
6. **루트 README.md 작성**. 구성:
   - 상단: "MS AI SW Academy (Microsoft × SKKU K-Digital Training) 이수 기록" + 기간 + 1~2줄 개요
   - **수료 증빙 블록**: Azure AI-900 자격증 이미지 + 수료증 이미지 (있으면). 없으면 "수료증 스캔 예정" 플레이스홀더
   - **6개 모듈 맵**: 테이블 (번호 · 모듈명 · 기간 · 핵심 개념 1줄 · 링크)
   - **Highlights** 섹션: 포트폴리오 관점에서 가장 보여주고 싶은 결과물 3~5개 직접 링크 + 2~3문장 설명 (후보 추천은 Claude Code가 Phase 1에서 제시)
   - **관련 프로젝트** 섹션: SOHOBI 레포 (`https://github.com/ProfessionalSeaweedDevourer/SOHOBI`) 링크 + "Academy에서 학습한 기술이 SOHOBI 팀 프로젝트의 기반이 됨" 한 줄
7. **`.gitignore` 점검**: `.env`, `*.pyc`, `__pycache__/`, `.ipynb_checkpoints/`, `venv/`, `.vscode/`(선택), 10MB+ CSV·PKL·H5
8. **LICENSE**: MIT 권장 (사용자 확인)

### Phase 2 산출물

- 정비된 디렉토리 트리
- 루트 + 6개 모듈 README
- `.gitignore`, `LICENSE`
- Dry-run 요약: 변경 파일 수, 삭제 파일 목록, 새로 만든 README 수, Critical/High 중 미해결 건

**commit 전 사용자 최종 확인 대기.** 전략(단일 PR / 모듈별 분리 / 단계별 분리)은 사용자에게 질문.

## Phase 3 — Highlights 심화 (별도 세션)

사용자가 "이 중 X·Y·Z를 Highlights로 끌어올리고 싶다"를 지정하면:

- 해당 결과물의 **독립 README** 작성 (문제 정의 → 접근 → 구현 → 결과 → 배운 점)
- 실행 재현 가능 여부 확인 (의존성 파일, 실행 커맨드 명시)
- 스크린샷·다이어그램이 유효하면 `assets/` 추가
- **수업 과제 응용 여부를 반드시 명시** (독자 혁신으로 읽히지 않도록)

## 절대 금지 사항

- 수업 슬라이드·교재·시험 문제 원본 **공개**. 의심 시 사용자 판단 받기
- `.env` · API 키 · 학번 · 개인 이메일 등 **민감정보 커밋** (history 포함)
- 저작권 모호한 외부 데이터셋을 **그대로 재배포**. 링크 또는 "출처:" 표기로 대체
- 학습 중 혼란·오류·미완 기록을 **"정리 명목으로 삭제"**. 실수와 수정 과정 자체가 가치 있으므로 보존. 공개 부적합 정보만 제거
- Eric의 허가 없이 **SOHOBI 레포의 private 브랜치·미공개 문서 내용 이식**
- **자기 감상·과장 표현**: "이 과정을 통해 크게 성장했습니다" 류 자제. 사실·수치 중심 서술

## 톤·스타일 가이드

- README 톤: 겸손하고 사실 중심. "~을 활용해 X 구현" 선호, "~을 통해 성장" 회피
- "수업 과제로 진행한", "~ 실습을 통해 학습" 식 **정직한 프레이밍**. "독자 연구"·"독립 프로젝트"로 확대 금지
- 한국어 맞춤법·띄어쓰기 검수
- 코드 파일 상단 주석: 한 줄 요약 + 모듈 번호 + 작성 일자
